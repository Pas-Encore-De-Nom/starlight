## Contexte

Le serveur GOAT comporte de nombreux plugins (XP, Jobs, Market, Factions, Claims…) qui ont tous besoin d'accéder et de modifier des données associées aux joueurs. Sans architecture partagée :

- Chaque plugin stockerait ses données séparément → duplication
- Les interactions entre plugins (ex: tuer un mob → XP + récompense Job + mise à prix PvP) nécessiteraient du couplage fort entre plugins
- Ajouter un nouveau plugin signifierait modifier tous les autres

## Décision

Utiliser une **architecture ECS (Entity Component System)** :

- Chaque joueur est une **entité**
- Les données (XP, balance, puissance faction…) sont des **composants** attachés à l'entité
- Les plugins interagissent via un **bus d'événements** plutôt que des appels directs

```
Player (Entity)
├── XpComponent         { total: 4200, sources: [...] }
├── BalanceComponent    { amount: 15000.0 }
├── FactionComponent    { factionId: "ALPHA", role: OFFICER }
└── JobComponent        { job: MINER, level: 12, dailyXp: 340 }
```

## Alternatives envisagées

- **Singletons par plugin** — Simple mais crée du couplage fort. Dès que deux plugins interagissent, ils doivent se connaître mutuellement. Rejeté car difficile à faire évoluer.
- **Base de données centrale avec accès direct** — Performances médiocres (I/O synchrone pour chaque action joueur). Rejeté.
- **Event system simple sans ECS** — Faisable mais sans structure de données commune : chaque plugin devrait interroger les autres pour accéder aux données joueur. Rejeté.

## Conséquences

### Positives

- **Découplage** : un nouveau plugin s'abonne aux événements du bus sans modifier les autres plugins.
- **Performance** : les données joueurs sont en mémoire, pas d'I/O synchrone.
- **Testabilité** : chaque composant peut être testé indépendamment.
- **Extensibilité** : ajouter un composant `CasinoComponent` ne touche à rien d'existant.

### Négatives / Trade-offs

- Courbe d'apprentissage pour les nouveaux contributeurs non familiers avec ECS.
- La persistance (sauvegarde sur disque) des composants doit être gérée explicitement.
- Le bus d'événements peut devenir un goulot d'étranglement si mal utilisé — surveiller avec le [plugin stress-test](/starlight/systemes/stress-test/).

<Aside type="conseil">
Pour tout nouveau composant ECS, documentez les événements qu'il émet dans la Javadoc de la classe. Cela permet aux autres développeurs de savoir à quoi s'abonner.
</Aside>

## Statut

**Acceptée** — Mars 2026
